# **Полный “язык ожиданий”**:

1. **Core axes** (спектры опыта)
2. **Interest modules** (жанровые модули с интенсивностью)
3. **Implementation modes** (как именно это допускается/проявляется)

Подробное описание осей лежит в папке `docs/axises`

---

# 1) Core axes (9 “главных осей”) 

Это не жанры. Это **как игрок хочет переживать игру**.
Шкала у каждой оси: **-2 -1 0 +1 +2** (или -3..+3) + `confidence`.

### A1. Тон

* **-**: тёплый / уютный / «в целом безопасно эмоционально»
* **+**: мрачный / тяжёлый / «эмоционально давит, но цепляет»
* зачем: задаёт базовую атмосферу и допустимый уровень “темноты”

### A2. Лёгкость и юмор

* **-**: почти без юмора / серьёзно
* **+**: юмор заметен / странности норм
* зачем: комедия — отдельная ось, а не часть тона (можно мрачно, но с юмором)

### A3. Давление и ощущение угрозы

* **-**: можно выдохнуть, опасность эпизодическая
* **+**: постоянная угроза, мир “давит”
* зачем: определяет общий уровень напряжения (даже без летальности)

### A4. Жёсткость последствий (летальность/ошибки)

* **-**: “кинематографично”, ошибки прощаются
* **+**: “жёстко”, ошибки дорого стоят
* зачем: отдельно от угрозы; можно “давит” без смертей, и наоборот

### A5. Стиль агентности

* **-**: планирование, подготовка, осторожность
* **+**: импровизация, риск, «по ситуации»
* зачем: влияет на дизайн сцен (головоломка/подготовка vs хаос/момент)

### A6. Структура истории

* **-**: свободное путешествие/песочница (мир реагирует)
* **+**: эпизоды/миссии с началом-концом (понятные “главы”)
* зачем: это главный структурный выбор кампании

### A7. Фокус опыта

* **-**: личные отношения/персонажи/диалоги
* **+**: мир/экспедиции/открытия/карта
* зачем: отличает “камера на людях” от “камера на мире”

### A8. Моральная текстура

* **-**: ясные правильные решения (комфорт)
* **+**: серые дилеммы и цена выбора
* зачем: фундаментально меняет ощущения от сюжета

### A9. Способ решать проблемы (социально ↔ действием)

* **-**: разговоры/сделки/убеждение/интриги
* **+**: действие/риск/схватки/прорыв
* зачем: это не “про бои”, а про привычный инструмент решения

> Если хочешь сжать до 8 осей — можно слить A3 и A4 в “интенсивность опасности”, но я бы оставил отдельно: это сильно повышает точность.

# 2) Interest modules (жанровые модули, интенсивность 0–3)

Это “специи”: **насколько игрок хочет, чтобы тема присутствовала**.
Шкала: `0=нет`, `1=фоном`, `2=заметно`, `3=в центре` + `confidence`.

## Рекомендованный набор модулей (10–12)

### M1. Детектив/загадки

* 0..3
* что значит: расследования, улики, скрытые связи, «понять почему»

### M2. Мистерия мира (тайна сеттинга)

* отличается от детектива: не “кто убийца”, а “что за мир и как он работает”

### M3. Политика/фракции

* интриги, союзы, влияние, большие последствия

### M4. Романтика (как тема)

* НЕ формат. Только “нужно ли вообще и насколько”

### M5. Хоррор/страх

* не обязательно мрак; это про “жутко/тревожно/неуютно”

### M6. Экспедиции/путешествия

* дороги, карты, маршруты, вылазки, смена локаций

### M7. Крафт/экономика/прогресс через добычу

* улучшения, ресурсы, обмен, создание предметов/решений

### M8. Большие “босс-сцены” / масштабные угрозы

* не обязательно часто, но если 2–3 — это “опорные эпизоды”

### M9. Социальная драма (отношения/конфликты внутри группы)

* отдельно от романтики: дружба, предательство, доверие, “семья партии”

### M10. Хитрость/авантюры/ограбления

* планы, проникновения, обман, “дело”

### M11. Быт/уют/“даунтайм”

* жизнь между опасностями: дом, праздники, разговоры у костра

### M12. Боёвка как частота (очень аккуратно)

* 0..3, но описывать не “бой”, а “частота острых столкновений”
* это модуль, а не core-ось, чтобы не превращать всё в механику

# 3) Implementation modes (как именно это допустимо) (лежат в файле `docs/axises/I.md`)

Это не “нравится/не нравится”. Это **формат разрешения**.

## I1. Романтика — форматы (набор флагов)

* `romance.pc_npc` (PC↔NPC)
* `romance.pc_pc` (PC↔PC)
* `romance.npc_npc` (NPC↔NPC фоном)
* `romance.explicitness` (0..2): намёками / сцены “в лоб” / “за кадром”
  (последнее лучше связывать с safety-частью, но как формат — полезно)

## I2. Внутрипартийный конфликт

* `party_conflict`: 0..2

  * 0: избегаем
  * 1: допустим мягко/ролево
  * 2: допустим остро, но с правилами

## I3. Уровень “потери персонажа”

* `character_loss`: 0..2

  * 0: почти невозможно
  * 1: возможно, но редко
  * 2: часть жанра

## I4. Границы сложности

* `rules_complexity`: 0..2

  * 0: максимально просто
  * 1: умеренно
  * 2: готовы к сложным штукам
    (для новичков это важно как UX-настройка)

# 4) Как это должно выглядеть в результате (UI)

**Страница результата = профиль ожиданий**, без названий типов.

## Блок A: Core profile

* список осей A1–A9
* для каждой: шкала с маркером + подписи полюсов
* рядом **confidence** (например, “уверенно / средне / нужно уточнить”)

## Блок B: Interest modules

* список M1–M12 как вертикальные “индикаторы интенсивности” (0..3)
* рядом confidence

## Блок C: Implementation modes

* чекбоксы/переключатели итогового состояния (что допустимо)

## Кнопки

* **Поделиться результатом** (ссылка/токен/копировать)
* **Хочу ещё поотвечать на вопросы** (продолжение с текущим state)

---

# 5) Оси и модули должны задаваться JSON-данными

Это правильная архитектура.

Минимальная схема (концептуально):

```json
{
  "axes": [
    {
      "id": "tone",
      "title": "Тон",
      "neg_label": "Уютный",
      "pos_label": "Мрачный",
      "scale": { "min": -2, "max": 2, "step": 1 }
    }
  ],
  "modules": [
    {
      "id": "mystery_detective",
      "title": "Детективность",
      "levels": ["нет", "фоном", "заметно", "в центре"]
    }
  ],
  "modes": [
    { "id": "romance.pc_npc", "type": "bool", "title": "Романтика PC↔NPC допустима" }
  ]
}
```

Движок не должен иметь зашитых смыслов — только работать с id и шкалами.

# 6) Когда “решение”: задаём новый вопрос или останавливаемся?

Канон:

1. **Система решает, что уже может показывать результат** (saturation):

   * по ключевым осям confidence ≥ порога
   * нет сильных конфликтов
   * “профиль стабилен”

2. **Но система не завершает автоматически.**
   Она спрашивает пользователя:

   > показать результат или уточнить ещё?

На result-странице кнопка “Хочу ещё поотвечать” запускает режим:

* выбирать вопросы по осям с низким confidence
* и по модулям, которые не определены, но важны (например, детектив/романтика)
