> Документ верхнего уровня.
> Описывает **цели, смысл, пользовательский путь и интерпретацию результата** адаптивного опросника.
>
> Детали интерфейса и визуального поведения описаны в:
> - *UI/UX Design Document — Adaptive Questionnaire*
>
> Детали логики, контрактов и алгоритмов описаны в:

# 1. Назначение системы

Адаптивный опросник предназначен для помощи игроку в формировании **осознанного понимания собственных игровых ожиданий**.

Система особенно ориентирована на пользователей, которые:
- никогда ранее не играли в НРИ;
- не знакомы с жанровыми терминами;
- затрудняются сформулировать предпочтения словами;
- колеблются между разными форматами опыта.

Опросник не классифицирует и не оценивает пользователя.

Он выполняет функцию **зеркала ожиданий**.

---

# 2. Философия продукта

## 2.1 Это не тест

Система не проверяет знания и не выносит вердиктов.

Она ведёт диалог, постепенно формируя гипотезу о предпочтениях игрока.

Подробное описание пользовательского ощущения «не-теста» см. в:
> UI/UX Design Document — раздел «Ключевая UX-метафора».

## 2.2 Адаптивность без искусственного интеллекта

Поведение системы может восприниматься как «умное», однако:

> система не использует машинное обучение или ИИ.

Все реакции формируются за счёт:
- детерминированных правил;
- весов осей;
- confidence-модели;
- выбора следующего вопроса по информационной ценности.

Техническая реализация описана в:
> Tech Design — State Machine и Question Selection.

---

# 3. Основные цели

## 3.1 Цели для игрока

- безопасно и интуитивно отвечать на вопросы;
- не чувствовать давления правильного выбора;
- постепенно лучше понять собственные ожидания;
- увидеть результат в понятной визуальной форме;
- иметь возможность уточнять профиль.

## 3.2 Цели для ведущего / организатора

- получить структурированное представление о предпочтениях игроков;
- использовать профиль как основу для обсуждения;
- снизить риск несоответствия ожиданий внутри группы.

---

# 4. Общий пользовательский путь

1. Пользователь попадает на страницу опросника.
2. Проходит адаптивный диалог с системой.
3. Система постепенно формирует профиль предпочтений.
4. При достижении достаточной определённости система предлагает показать результат.
5. Пользователь принимает решение:
   - посмотреть результат;
   - или продолжить уточнение.
6. На странице результата пользователь может:
   - изучить профиль;
   - поделиться им;
   - вернуться к вопросам.

Важно:
- путь не линейный;
- количество вопросов не фиксировано;
- пользователь сохраняет контроль над завершением.

---

# 5. Обработка каждого ответа

Каждый ответ пользователя проходит единый логический цикл.

## 5.1 Фиксация

Система сохраняет:
- идентификатор вопроса;
- выбранный вариант или значение;
- тип взаимодействия;
- временную метку.

Ответ не интерпретируется как правильный или ошибочный.

## 5.2 Преобразование в наблюдения

Ответ преобразуется в набор наблюдений:
- изменение значений осей (score);
- повышение или снижение уверенности (confidence);
- выявление противоречий;
- установку дискретных тегов.

Механика описана в:
> Tech Design — Axes and Confidence Model.

## 5.3 Обновление состояния

После каждого ответа пересчитывается текущее состояние профиля:
- значения осей;
- уровень их изученности;
- наличие конфликтов;
- относительное положение кандидатных профилей.

## 5.4 Оценка насыщенности информации

Система оценивает:
- достаточно ли изучены ключевые оси;
- сохраняется ли неопределённость;
- существует ли близкая альтернатива.

---

# 6. Решение: продолжать или остановиться

## 6.1 Принцип двух уровней решения

В системе действует разделение ответственности:

1. **Система** определяет, когда информации достаточно для показа результата.
2. **Пользователь** определяет, хочет ли он завершить диалог.

Когда система достигает информационной насыщенности, она сообщает:

> «Кажется, я уже достаточно хорошо понял тебя.
> Показать результат — или хочешь уточнить ещё немного?»

Это правило является каноническим для всей системы.

---

# 7. Продолжение опроса после результата

Результат не является финальной точкой.

На странице профиля всегда присутствует действие:

> **«Хочу ещё поотвечать на вопросы»**

При выборе этого действия:
- текущий state сохраняется;
- ранее заданные вопросы не повторяются;
- приоритет получают оси с низким confidence;
- система переходит в режим углубления.

Таким образом, результат становится новой точкой обзора, а не завершением.

---

# 8. Результат опроса

Результатом является **профиль игровых ожиданий**, а не классификация личности.

Система не присваивает названий ролей или типов.

Она отображает структуру предпочтений.

---

# 9. Представление результата

## 9.1 Общий принцип

Результат должен быть:
- понятным без пояснений;
- визуально читаемым;
- неоценочным.

## 9.2 Отображение осей

Пользователю показываются:
- все оси ожиданий;
- положение на каждой оси;
- человеческое описание полюсов.

Например:
- Медленный ↔ Динамичный
- Безопасный ↔ Опасный
- Лёгкий ↔ Мрачный

Названия и формулировки осей задаются данными.

## 9.3 Отображение уверенности

Для каждой оси отображается степень уверенности:
- хорошо изучена;
- изучена частично;
- требует уточнения.

Это напрямую отражает confidence.

Подробная модель confidence описана в:
> Tech Design — Axes and Confidence Model.

## 9.4 Многоуровневость результата

Результат имеет два уровня:

1. **Основной слой** — краткий визуальный профиль.
2. **Расширенный слой** — подробное объяснение по осям и уверенности.

Расширенное объяснение раскрывается по желанию пользователя.

---

# 10. Шеринг результата

## 10.1 Назначение

Результатом можно делиться:
- с ведущим;
- с группой;
- с друзьями;
- для личного сохранения.

Цель шеринга — коммуникация, а не сравнение.

## 10.2 Состав передаваемых данных

При шеринге передаются:
- значения осей;
- confidence по осям;
- краткое текстовое описание профиля.

Не передаются:
- история ответов;
- конкретные вопросы;
- персональные данные.

Это делает результат безопасным и переносимым.

---

# 11. Конфигурационность системы

## 11.1 Оси как данные

Все оси определяются через JSON-конфигурацию.

Движок не содержит смыслов.
Он оперирует абстрактными измерениями.

Это позволяет:
- изменять язык ожиданий;
- адаптировать систему под другие домены;
- расширять модель без переписывания логики.

## 11.2 Вопросы как данные

Все вопросы описываются декларативно.

Они определяют эффекты, но не содержат алгоритмов.

Выбор следующего шага осуществляется движком.

---

# 12. Итоговый принцип

Система никогда не говорит:

> «Ты вот такой человек».

Она говорит:

> «На основе твоих ответов сейчас твои ожидания выглядят вот так».

И всегда оставляет пространство для изменения и уточнения.
